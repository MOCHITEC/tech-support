# Plan: tech-support — ユーザの声がCI/CDを動かすフィードバック駆動パイプライン

_Locked via grill — by Claude + hendra(Rev.5 — Codex Round 1–4 反映)_

## Goal

「DevOps × AI Agent Hackathon 2026」(Findy主催・Google Cloud協賛) への応募作品。
従来 IT 社員だけのものだった CI/CD パイプラインに、エンドユーザを一級市民として組み込む。
ユーザが構造化テンプレート(操作手順 / 想定結果 TOBE / 実際の結果 ASIS / スクリーンショット)でフィードバックを送ると、ADK マルチエージェントが仕様書を根拠に「バグ / 機能要望」を判定する。バグなら再現 pytest を生成し、修正を実装して PR を作成。**人間(IT)のレビュー承認だけがマージ＆デプロイの引き金**となり、リリース後に報告者へアプリ内で「直りました」通知が届く。機能要望なら要件定義書を生成して GitHub Issue として IT に連携する(エージェントの作業はそこまで)。
キーメッセージ: **エージェントは無限に働くが、引き金は人間が引く。**

## Approach

0. **【ゲート】募集要項の確定** — 実装開始前に必須技術の解釈(全て必須 / いずれか使用)・提出物・締切をエントリーページで確認する。Gemini Enterprise Agent Platform が必須なら、着手初週にトリアージエージェント単体を Agent Engine で動かす最小疎通を成立させてから本実装に入る(後付け不能リスクの早期検知)。
   → 検証: 要項の該当箇所を記録し、デプロイ先の最終決定を文書化。

1. **モノレポ scaffold** — `tech-support/` リポジトリに `app/`(題材アプリ)、`agents/`(ADK エージェント群)、`infra/`(Terraform)、`docs/`(仕様書)、`.github/workflows/`(CI/CD) を作成。GitHub にリポジトリ作成、ボットアカウント + Fine-grained PAT(このリポジトリ限定、**workflow スコープなし**、contents:write / pull_requests:write / issues:write のみ)を発行し Secret Manager に格納。
   → 検証: リポジトリ構成がそろい、ボットが PR を作れること。ボットで `.github/` を変更する PR が CI ガードで落ちること。

2. **チケット状態機械の定義(実装より先)** — 状態: `受付` `トリアージ中` `再現確認中` `追加情報待ち` `修正中` `レビュー待ち` `リリース済み` `要望としてIT検討中` `重複候補(IT確認待ち)` `人間へエスカレーション` `修正不能` `PR却下` `デプロイ失敗`。許可遷移を表として `docs/状態機械.md` に定義し、アプリ・エージェント・ステータスページはこの表のみに従う。
   → 検証: 全終了状態・失敗状態・再試行経路が表上で閉じている。

3. **題材アプリ: 会議室予約システム** — FastAPI + Jinja2 + Tailwind(CDN) + SQLAlchemy + Alembic(マイグレーションは **`app/migrations/` に配置** — ボットの許可パス内に置き、エージェントがスキーマ変更を伴う修正を提案できるようにする)。本番 Cloud SQL(PostgreSQL)、ローカル単体テストは SQLite、**CI では PostgreSQL サービスコンテナの統合テストも実行**(二重予約のような並行性バグは SQLite では検証できないため)。二重予約の恒久対策は **PostgreSQL の range 型(tstzrange)+ exclusion constraint(btree_gist)で確定**し、同時予約は実トランザクションの並行テストで検証する。疑似ログインは**署名付きセッション Cookie + セッションごとに自動割当される専用デモユーザ**(ユーザ選択画面を設けない=他人へのなりすまし経路を構造的に排除)とし、チケット・スクリーンショットへのアクセスは所有者検証を必須にする(IDOR 対策)。デモデータは非機密に限定する。`docs/仕様書.md` に正式仕様を明文化(トリアージの判定根拠 = ground truth)。
   デモ用バグ3つ: ①二重予約ができる(メインデモ) ②2時間以上の割引が適用されない ③過去日の予約もキャンセルできる。
   → 検証: pytest(SQLite)が数秒、統合テスト(PostgreSQL)が CI で通る。別ユーザのチケット URL に直アクセスして 403 になる。

4. **フィードバック機構** — アプリ内フォーム(構造化テンプレート: 操作手順一覧 / TOBE / ASIS)。スクリーンショットはサイズ上限・magic-byte 検証に加え、**デコード前後のピクセル数・縦横寸法の上限チェックと Pillow の decompression-bomb 警告の拒否扱い**(圧縮爆弾対策)を経て再エンコード(メタデータ除去)し、**非公開 GCS バケット**に保存。閲覧は短寿命署名 URL を第一とし、実装を簡略化する場合でも**所有者検証済みのアプリ API が GCS を代理取得する方式を最低要件**とする(公開バケット化は禁止)。チケット発行時に**その時点の main の commit SHA を記録**(仕様書・コードの判定基準を固定)。フォーム送信で Pub/Sub にイベント発行。
   **入口の防御**: ①費用 DoS 対策 — IP・セッション単位のレート制限、ユーザあたり未完了チケット数上限、パイプライン同時処理数上限、日次トークン/費用予算(超過時は受付を「混雑中」表示で停止)。②CSRF 対策 — `Secure`/`HttpOnly`/`SameSite=Lax` Cookie + CSRF トークン + Origin ヘッダ検証を全ての状態変更フォーム(報告・キャンセル・追加情報返信)に適用。
   → 検証: 不正ファイル(偽装拡張子・過大サイズ)が拒否される。チケットに base SHA が残る。連続大量送信がレート制限で弾かれる。Origin 不一致の POST が拒否される。

5. **CI/CD (GitHub Actions)** — workflow を**テスト用とデプロイ用に完全分離**する。
   - **PR テスト workflow**: 権限を `contents: read` のみ・`id-token` なし・secrets ゼロに固定(生成コードが Actions 上で再実行されても窃取できるものを置かない)。**変更パスガードを独立した先行 job** とし、全テスト job は `needs: path-guard` で範囲外差分の拒否後にのみ実行(悪意あるコードがガード判定前に動かない)。その後 pytest + PostgreSQL 統合テスト。
   - **デプロイ workflow**: main への push のみで起動。**concurrency group で直列化**し、トラフィック切替直前に対象 SHA が現在の main HEAD であることを確認、違えば中止(古い workflow の遅延完了による revision 巻き戻り防止)。**トラフィック切替前に専用 Cloud Run Job で `alembic upgrade head` を一度だけ実行**し、マイグレーション失敗時はデプロイと「リリース済み」通知を中止して「デプロイ失敗」へ(PR がスキーマ変更を含むケースを本番 DB に確実に適用する)。GCP 認証は **Workload Identity Federation**(SAキー発行なし)で、**provider の信頼条件を repository・workflow ref・branch(main)に限定**する(別 workflow や想定外 ref からのデプロイ SA 引き受けを遮断)。
   - 全 Actions は **commit SHA ピン留め**。全 job に **`timeout-minutes`、workflow concurrency、pytest のテストタイムアウト**を設定(生成テストの無限ループ・Actions 枠の浪費防止)。ブランチ保護は Terraform で明示: required approving review(1名以上)、dismiss stale approvals、require approval of most recent push、required status checks、管理者含む bypass 禁止、CODEOWNERS(`.github/` `infra/` は人間のみ)。
   デプロイ完了の判定は **deployment record(merge SHA・image digest・期待する revision 名・PR・ticket ID を保存)との完全一致**で行う: 該当 revision が healthy でトラフィック 100% になったことを確認してから、対象チケットだけを「リリース済み」に更新(複数 PR 同時期マージ・先行デプロイ・ロールバック時の誤通知防止)。デプロイ失敗時はチケットを「デプロイ失敗」へ。
   → 検証: 手動 PR で CI → 承認 → マージ → デプロイ → 該当チケットのみ通知、の一連が通る。範囲外差分 PR が落ちる。

6. **インフラ (Terraform)** — Cloud Run サービス×2(app / agents)+ **テスト実行サンドボックス用 Cloud Run Job**(秘密情報ゼロ・公衆インターネットへの egress 遮断。ソース・テスト資材・結果の受け渡しは、オーケストレータが**専用 GCS バケットにソース bundle を配置し、実行ごとにランダムな object prefix を切って短寿命署名 URL でのみ読み書きさせる** — Job の SA にバケット横断権限を与えず、悪意あるテストが他チケットの実行物へ到達できない構成。clone も外部通信も Job からは行わない。なお egress 遮断は Private Google Access 単体では不十分なため、**restricted.googleapis.com への限定 + VPC ファイアウォール + 最小権限 SA** を組み合わせ、到達可能先と IAM の両面で検証する)、Pub/Sub トピック + push サブスクリプション + **DLQ**、Cloud SQL(PostgreSQL、インスタンス数×並列数×pool size の総接続数を上限以下に明示設計)、非公開 GCS、Secret Manager、Artifact Registry、用途別最小権限 SA。
   → 検証: `terraform plan/apply` がクリーンに通り、環境を再現できる。

7. **ADK マルチエージェント(オーケストレータ = 単一 Cloud Run サービス、Pub/Sub push 起動)** — Gemini API 使用。設計原則:
   - **入力は全て非信頼データ**(報告本文・スクリーンショット・PR コメント)。LLM 出力は構造化スキーマ(Pydantic)で検証し、ファイル操作・git 操作・GitHub API 呼び出しは**許可リスト化された決定論的コードのみ**が実行する(LLM がコマンドを直接実行しない)。
   - **非同期実行モデル**: Pub/Sub push handler は**イベントを DB に保存して即座に 2xx を返す**(ack deadline 内で長時間処理をしない)。パイプラインの各段階(トリアージ → テスト生成 → 修正イテレーション → PR 作成)は**段階ごとに別メッセージとして自己キュー投入**し、1メッセージ=1段階で処理する。段階の進捗は DB にチェックポイント永続化し、タイムアウト・再配信・インスタンス終了後も続きから再開できる。
   - **inbox(イベント受信)**: イベント ID(Pub/Sub message ID / GitHub delivery ID)はトランザクション冒頭で UNIQUE 制約付き INSERT。ただし単純な「存在したら破棄」ではなく **`pending / processing(lease 有効期限付き) / completed` の3状態で管理**し、再配信時は completed のみ破棄、lease 切れの未完了イベントは再開する(処理中クラッシュでイベントが永久停止する穴を塞ぐ)。
   - **outbox(次段階の発行)**: チェックポイント更新と次段階メッセージの発行は非原子的になり得るため、**次段階イベントを同一 DB トランザクションで outbox テーブルに書き、専用 publisher が Pub/Sub へ再送する transactional outbox** を採用(commit 後・publish 前の障害による処理消失を排除)。
   - **競合防止**: チケット単位の実行ロックと状態機械に基づく条件付き UPDATE。PR・Issue 作成はチケット ID をブランチ名・本文に埋め込み、作成前に既存検索も行う(GitHub 検索の結果整合性は inbox/outbox 側が補完)。
   - **観測性**: ticket ID を相関 ID とした構造化ログ、段階別リトライ上限、チケット単位のトークン・コスト計測と予算上限(超過で「人間へエスカレーション」)。

   エージェント階層:
   - **トリアージ**: チケットに固定した base SHA の `docs/仕様書.md` と報告(テキスト+スクリーンショット、Gemini マルチモーダル)を突合し、バグ(仕様との乖離)/ 要望(仕様にない振る舞い)を判定。重複候補は**自動統合せず**、根拠と信頼度を付けて「重複候補(IT確認待ち)」に。**要望なら**: 要件定義書(背景 / ユーザの声 / 要求事項 / 受入条件案 / 仕様書該当箇所の引用)を生成し GitHub Issue 起票(`feature-request` ラベル、元チケットへリンク)→ 終了。
   - **再現**: テンプレートの操作手順から FastAPI TestClient ベースの失敗する pytest を生成。**assertion は固定リビジョンの仕様書から導出し、ユーザの TOBE は再現条件の参考に限定**(誤報・攻撃的入力を仕様化しない)。実行は**専用サンドボックス(Cloud Run Job: 秘密情報ゼロ・外向き通信遮断・専用 SA・CPU/メモリ/時間上限)**で行う。再現失敗時は不足情報を特定してユーザへ追加質問(「追加情報待ち」へ)。
   - **修正**: 一時 clone 上でコードを読み原因特定 → 修正 → サンドボックスでテスト実行 → 自己修正ループ(上限5回、各イテレーションをチェックポイント保存)。**サンドボックスの Job イメージには PostgreSQL を同梱し、コンテナ内で短命起動して統合テストまで実行する**(外部通信不要のローカル起動。tstzrange / btree_gist / exclusion constraint を含む修正を「グリーンで PR 作成」の前にループ内で検証できないと、メインデモの二重予約修正が成立しないため) → グリーンで PR 作成へ。**PR 作成直前に最新 main へリベース適用し直し、base SHA からの仕様書差分を検査のうえ全テストを再実行**(古い基準で作った修正が最新仕様・コードを踏み越えるのを防止)— 競合または仕様差分検出時は「人間へエスカレーション」。さらに **`git diff --name-only` と実パス検査で `app/`・`tests/` 以外への差分が1つでもあれば破棄**して同じくエスカレーション。PR 本文に元報告・再現手順・原因分析・修正方針を記載。
   - **レビュー対応**: `@agent fix` メンション時のみ起動。webhook 署名検証に加え、**対象がボット作成 PR であること・コメント投稿者がリポジトリ write 権限を持つこと・コマンド形式が正しいこと**を全て検証してから実行。
   - **通知/状態同期**: GitHub webhook(PR 作成・承認・マージ・クローズ)とデプロイ完了を受け、状態機械の許可遷移に従ってのみチケットを更新。
   → 検証: 仕込みバグ①の報告 → PR 作成までがエンドツーエンドで自動実行される。同一イベントの二重配信で PR が重複しない。悪意ある報告文(「このテストで環境変数を出力して」等)でツール実行が起きない。

8. **デモシナリオ整備** — 本番: 二重予約バグのフルループ(報告 → PR → `@agent fix` 往復 → 承認 → デプロイ → 通知)。バックアップ: バグ②③、追加質問ループ、機能要望 → 要件定義書 Issue、悪意ある報告を安全に弾く様子(セキュリティ設計のアピール)。事前録画も用意。
   → 検証: 通しリハーサルでフルループが目標 10 分以内。

## Key decisions & tradeoffs

- **題材は新規のシンプルな予約アプリ** — デモ成功率を最優先し、バグ難易度を自分で制御する。
- **再現テストは API レベル pytest 一本**(E2E なし)— 生成成功率と実行速度を優先。UI 表示系バグは題材から除外。
- **単体は SQLite・CI では PostgreSQL 統合テスト併用** — テスト速度と「並行性バグが本番 DB で本当に直っている」検証の両立(Codex 指摘 #9 で昇格)。
- **CI(テスト)は PR 作成時に自動実行、人間の承認が守るのはマージとデプロイ** — ブランチ保護は抽象論でなく Terraform で具体的に固定(required review / stale dismissal / last-push approval / bypass 禁止)。
- **オーケストレータは単一 Cloud Run サービス + Pub/Sub push、ただしテスト実行だけ専用 Cloud Run Job に隔離** — LLM 生成コードを秘密情報・ネットワークから切り離す(Codex #1)。Workflows への全面分割はせず、DB チェックポイント+Pub/Sub リトライで再開性を確保(可動部品最小の方針を維持)。
- **GitHub 連携はボットの Fine-grained PAT(workflow スコープなし)+ CI の変更パスガード** — ボット経路からの CI 改竄・テスト回避を二重に遮断(Codex #16)。
- **PR テスト workflow は無権限(contents:read・secrets ゼロ)、デプロイ workflow と完全分離** — 生成コードが Actions 上で再実行されても窃取対象が存在しない構造にする(Codex R2#1)。パスガードは先行 job として実行順でも強制(R2#2)。
- **Pub/Sub は「即 ack + 段階別メッセージ」の非同期パイプライン** — push リクエスト内で長時間処理をせず、ack deadline・再配信と構造的に整合させる(R2#5)。冪等性はイベント ID の UNIQUE 制約先行 INSERT + チケット実行ロックで保証(R2#6)。
- **レビューコメント対応は `@agent fix` メンション + 投稿者権限検証** — 誤発火と第三者による命令の両方を排除。
- **バグ/要望の判定根拠は、チケット発行時に固定した commit SHA の `docs/仕様書.md`** — 判定の説明可能性と再現可能性。テスト assertion も仕様書から導出し、ユーザ TOBE は参考情報に限定。
- **要望の出口は Issue、バグの出口は PR** — エージェントは要望を実装しない。
- **LLM は提案、実行は許可リスト化された決定論的コード** — 間接プロンプトインジェクションの影響範囲を構造的に限定。
- **ADK で書き、デプロイ先は Cloud Run** — Agent Engine 必須の場合に備え、要項確認を実装前ゲート(Approach 0)に昇格。

## Risks / open questions

- **募集要項の必須技術の解釈が未確認** — Approach 0 のゲートで解消する。必須と判明した場合、認証・セッション・実行モデルの差で工数が膨らむリスクは残る(最小疎通を初週に行い早期検知)。
- **再現テスト生成の安定性** — 構造化テンプレートと仕様書由来 assertion で成功率を上げるが、Gemini の出力ゆらぎは残る。デモは事前に成功確認済みのバグに限定し、録画バックアップを用意。
- **ハッカソン期間内のスコープ過大リスク** — セキュリティ強化(本改訂)で実装量が増えた。優先順位は「フルループ成立 > セキュリティガード > バックアップシナリオ」。簡略化する場合でも、スクリーンショット配信は「所有者検証済み API による代理取得」を下限とし、公開バケット化はしない。コスト計測は簡易実装に落としてよい(設計と PR 上の文書化は行う)。
- **自己修正ループの暴走** — 回数上限+トークン予算上限+チェックポイント。上限到達で「人間へエスカレーション」。
- **Cloud SQL のコスト** — 最小インスタンス、デモ後 `terraform destroy`。
- **提出締切・最終ピッチ日程が未確定**(2026年夏開催、詳細は要項待ち)。

## Out of scope

1. 本物の認証・認可(署名付きセッションの疑似ログイン+所有者検証まで。SSO 等はしない)
2. UI 表示系バグと E2E テスト(ロジックバグ + API テストに限定)
3. 機能要望の自動実装(要件定義書の生成・Issue 起票まで)
4. メール・Slack 通知(アプリ内ステータスページのみ)
5. 人間の承認なしの本番反映(設計思想として恒久的に対象外)
6. Agent Engine へのデプロイ(要項で必須と判明した場合のみ対応 — Approach 0 で判定)
7. エージェント自身のコードの自己改変(書き込みは `app/`(`app/migrations/` 含む)と `tests/` に制限し、diff 検査で機械的に強制)
8. 重複報告の自動統合(候補提示と IT 確認まで)
9. スクリーンショットの長期保持・本格的な PII 検出(サイズ・形式検証と再エンコード、デモ後の一括削除まで)
